Catches the common runtime mistake of forgetting that the base attribute on @ResourcePath is a validation-time prefix only. At runtime the annotated parameter holds just the suffix supplied by the caller - the method body is responsible for prepending the base before actually loading the resource.

Sink check

Warns when a @ResourcePath(base = "X") String param is passed directly into a known resource-loading call without the X/ prefix. Offers a quick-fix that wraps the argument as "X/" + param. 

void loadResource(@ResourcePath(base = "templates") String fileName) {
    // Warned - at runtime, fileName is just "index.html", not "templates/index.html"
    getClass().getResourceAsStream(fileName);

    // Quick-fix rewrites to:
    getClass().getResourceAsStream("templates/" + fileName);
}

Recognised sinks:

  • Class.getResource, Class.getResourceAsStream
  • ClassLoader.getResource, getResourceAsStream, getResources, resources, and their getSystem* variants
  • Path.of, Paths.get
  • Constructors: File, FileInputStream, FileOutputStream, FileReader, FileWriter, RandomAccessFile

Forwarding check

Warns when a @ResourcePath(base = "X") parameter is passed as the argument to another parameter annotated @ResourcePath(base = "Y") where X != Y. The two paths disagree about what prefix the string represents, so the value will fail validation in one of the methods. 

void outer(@ResourcePath(base = "assets") String assetName) {
    // Warned - outer's base is "assets", loadTemplate expects "templates"
    loadTemplate(assetName);
}

void loadTemplate(@ResourcePath(base = "templates") String name) { ... }

Special cases:

  • Source has base, target has none - quick-fix is offered (add the prefix).
  • Source has none, target has base - warned without a quick-fix (direction is ambiguous).
  • Both have different non-empty bases - warned without a quick-fix.

Conservatively skipped

Non-literal expressions, arguments already wrapped in concatenation, values coming through intermediate variables, and sinks outside the recognised list never produce a warning - the inspection errs on the side of silence to avoid false positives.

Settings

  • Warn when ... passed to a resource-loading call without the prefix - toggles the sink check.
  • Warn when ... forwarded to another parameter with a mismatched base - toggles the forwarding check.
  • Highlight for sink-check violations - severity applied to sink warnings.
  • Highlight for forwarding-mismatch violations - severity applied to forwarding warnings. Independent from the sink severity.
  • Excluded file patterns - glob patterns for files the inspection skips entirely (e.g. **/generated/**).
